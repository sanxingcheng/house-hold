# 迭代四详细设计

## 一、迭代目标与范围

### 1.1 目标

在迭代三（财富管理）及 3.5（Redisson / 分布式锁 / Session 管理）基础上，完成系统级**优化与完善**：

- 接入 Spring Boot Actuator + Micrometer/Prometheus，暴露健康检查与指标监控端点
- 配置 Logback 结构化日志，区分 dev/prod 环境，并增加请求链路追踪（MDC）
- 使用 Spring Scheduler + Redisson 分布式锁实现夜间批量快照定时任务
- 引入 ShardingSphere-JDBC 对 `account` 表按 `user_id` 进行水平分表（4 张物理表）
- 前端全面引入 Element Plus 组件库 + ECharts 图表库，替换手写 CSS，统一设计语言
- 前端增加 loading 状态、消息通知、路由过渡动画等 UX 优化
- 前端 logout 调用后端登出接口，实现服务端 Session 失效

### 1.2 范围

| 类别       | 内容                                                                                                       |
|------------|------------------------------------------------------------------------------------------------------------|
| **监控**   | Actuator 端点配置、Micrometer Prometheus 指标暴露、自定义健康检查                                           |
| **日志**   | Logback 结构化配置（dev 彩色控制台 / prod JSON）、请求日志 Filter（MDC traceId + userId）、Gateway 访问日志 |
| **定时任务** | SnapshotScheduler 每日凌晨 2:00 批量快照，Redisson 分布式锁防多实例重复执行                                |
| **分表**   | ShardingSphere-JDBC 对 `account` 表按 `user_id % 4` 分为 `account_0`~`account_3`                           |
| **前端**   | Element Plus 组件库替换所有手写表单/卡片/表格/对话框；ECharts 替换 Canvas 图表；UX 优化                    |

### 1.3 不纳入本迭代

- 分库（仅分表，同一数据库内）
- Grafana 看板搭建（Prometheus 端点就绪，Grafana 为运维层面自行对接）
- ELK / 日志采集平台集成
- 前端国际化（i18n）
- 前端单元测试（Vitest）

### 2.3 Logback 结构化日志

**日志格式（dev）：**


**日志格式（prod / JSON）：**
```json
{"timestamp":"...","level":"INFO","logger":"...","traceId":"...","userId":"...","message":"..."}
```

### 2.4 请求日志 Filter（`house-hold-common`）

**类名：** `com.household.common.filter.RequestLoggingFilter`

**职责：**
- 拦截所有 HTTP 请求
- 生成 `traceId`（UUID 短格式）放入 MDC
- 从 `X-User-Id` 请求头读取 `userId` 放入 MDC
- 在请求完成后记录：`method`、`uri`、`status`、`duration(ms)`
- 通过 `@ConditionalOnWebApplication(type = SERVLET)` 仅在 Servlet 环境生效（不影响 Gateway WebFlux）

```java
@Component
@ConditionalOnWebApplication(type = ConditionalOnWebApplication.Type.SERVLET)
public class RequestLoggingFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest req, HttpServletResponse res, FilterChain chain) {
        String traceId = UUID.randomUUID().toString().substring(0, 8);
        MDC.put("traceId", traceId);
        MDC.put("userId", req.getHeader("X-User-Id"));
        long start = System.currentTimeMillis();
        try {
            chain.doFilter(req, res);
        } finally {
            long duration = System.currentTimeMillis() - start;
            log.info("{} {} → {} ({}ms)", req.getMethod(), req.getRequestURI(), res.getStatus(), duration);
            MDC.clear();
        }
    }
}
```

### 2.5 Gateway 访问日志

**类名：** `com.household.gateway.filter.AccessLogGlobalFilter`

**职责：**
- 作为 `GlobalFilter`（Ordered = -1），在请求进入和响应返回时记录
- 日志内容：`method`、`path`、`routeId`、`statusCode`、`duration(ms)`
- 使用 `reactor.util.context.Context` 传递 traceId

## 三、后端详细设计 — 定时快照任务

### 3.1 启用调度

在 `HouseHoldWealthApplication` 上添加 `@EnableScheduling`。

### 3.2 SnapshotScheduler

**包路径：** `com.household.wealth.scheduler.SnapshotScheduler`

**核心逻辑：**

```
@Scheduled(cron = "0 0 2 * * ?")     // 每日凌晨 2:00
void nightlySnapshot():
  1. 尝试获取 Redisson 分布式锁 "household:lock:snapshot-scheduler"（waitTime=0, leaseTime=30min）
  2. 若未获取到锁 → 说明其他实例正在执行，直接返回
  3. 查询所有有账户的 userId: accountRepository.findDistinctUserIds()
  4. 对每个 userId 查询 familyId，调用 snapshotService.triggerSnapshot(userId, familyId)
  5. 记录日志：总用户数、成功数、失败数、耗时
  6. finally 释放锁
```

**可配置项（`application.yml`）：**
```yaml
snapshot:
  scheduler:
    enabled: true
    cron: "0 0 2 * * ?"
```

使用 `@ConditionalOnProperty(name = "snapshot.scheduler.enabled", havingValue = "true", matchIfMissing = true)` 控制。

### 3.3 Repository 扩展

`AccountRepository` 新增：
```java

```

## 四、后端详细设计 — ShardingSphere 分表

### 4.1 分表策略

| 项目         | 值                                       |
|--------------|------------------------------------------|
| 逻辑表名     | `account`                                |
| 物理表名     | `account_0`, `account_1`, `account_2`, `account_3` |
| 分片键       | `user_id`                                |
| 分片算法     | `INLINE`: `account_$->{user_id % 4}`    |
| 影响模块     | `house-hold-wealth`（仅此模块访问 account 表） |

其他表（`wealth_snapshot`、`user_base`、`family`、`family_member_role`）不分表。

### 4.2 依赖

root `pom.xml` — `dependencyManagement` 新增：

```xml
<dependency>
    <groupId>org.apache.shardingsphere</groupId>
    <artifactId>shardingsphere-jdbc</artifactId>
    <version>${shardingsphere.version}</version>
</dependency>
```

`house-hold-wealth/pom.xml` 新增：

```xml
<dependency>
    <groupId>org.apache.shardingsphere</groupId>
    <artifactId>shardingsphere-jdbc</artifactId>
</dependency>
```

### 4.3 配置（`application.yml`）

由于 ShardingSphere-JDBC 接管 DataSource，需将原始 `spring.datasource` 替换为 ShardingSphere 配置：

```yaml
spring:
  datasource:
    driver-class-name: org.apache.shardingsphere.driver.ShardingSphereDriver
    url: jdbc:shardingsphere:classpath:shardingsphere.yaml
```

独立配置文件 `shardingsphere.yaml`：

```yaml
dataSources:
  ds0:
    dataSourceClassName: com.zaxxer.hikari.HikariDataSource
    driverClassName: com.mysql.cj.jdbc.Driver
    jdbcUrl: jdbc:mysql://127.0.0.1:3306/household?useUnicode=true&characterEncoding=utf8&serverTimezone=Asia/Shanghai
    username: root
    password: Admin@123

rules:
  - !SHARDING
    tables:
      account:
        actualDataNodes: ds0.account_$->{0..3}
        tableStrategy:
          standard:
            shardingColumn: user_id
            shardingAlgorithmName: account-inline
    shardingAlgorithms:
      account-inline:
        type: INLINE
        props:
          algorithm-expression: account_$->{user_id % 4}

props:
  sql-show: false
```

### 4.4 DDL 迁移脚本

```sql
-- 创建 4 张分表（结构与原 account 表一致）
CREATE TABLE IF NOT EXISTS account_0 LIKE account;
CREATE TABLE IF NOT EXISTS account_1 LIKE account;
CREATE TABLE IF NOT EXISTS account_2 LIKE account;
CREATE TABLE IF NOT EXISTS account_3 LIKE account;

-- 如果需要从原 account 表迁移存量数据：
-- INSERT INTO account_0 SELECT * FROM account WHERE user_id % 4 = 0;
-- INSERT INTO account_1 SELECT * FROM account WHERE user_id % 4 = 1;
-- INSERT INTO account_2 SELECT * FROM account WHERE user_id % 4 = 2;
-- INSERT INTO account_3 SELECT * FROM account WHERE user_id % 4 = 3;
```

### 4.5 JPA 调整

- `Account` 实体 `@Table(name = "account")` 保持不变，ShardingSphere 透明路由
- JPA `ddl-auto` 改为 `none`（ShardingSphere 管理的表不再自动建表）
- `wealth_snapshot` 等非分片表仍可正常使用

### 4.6 查询影响分析

| 查询方法           | 是否携带分片键 | 路由方式   | 性能影响 |
|--------------------|:--------------:|:----------:|:--------:|
| `findByUserId()`   | 是             | 精确路由   | 无影响   |
| `findById()`       | 否             | 广播查询   | 4 表扫描 |
| `findByFamilyId()` | 否             | 广播查询   | 4 表扫描 |
| `findDistinctUserIds()` | 否        | 广播 + 合并 | 4 表扫描 |

`findByFamilyId()` 为家庭汇总使用，调用频率低，广播查询可接受。

### 5.5 UX 优化

| 优化项         | 实现方式                                                       |
|----------------|----------------------------------------------------------------|
| 加载状态       | `v-loading` 指令覆盖数据容器                                   |

### 5.6 前端登出修复

**问题：** 当前 `authStore.logout()` 仅清除 localStorage，未通知后端失效 Session。

**方案：**
1. `api/auth.ts` 新增 `logout()` 函数：`request.post('/auth/logout')`
2. `stores/auth.ts` 的 `logout()` 改为 async，先调后端接口再清理本地状态
3. 错误静默处理（网络异常时仍完成本地清理）
