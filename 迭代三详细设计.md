# 迭代三详细设计

## 一、迭代目标与范围

### 1.1 目标

在迭代二（用户与家庭管理）基础上，新建独立微服务 **house-hold-wealth**，完成**家庭财富管理**全闭环：

- 个人账户 CRUD（支持多种账户类型与货币）
- 个人/家庭资产实时汇总
- 每次账户变更自动生成快照，支持历史资产趋势查询
- 前端新增账户管理页与资产趋势图页，概览页接入真实财富数据
- wealth-service 接入 Caffeine + Redis 三级缓存，覆盖账户列表、汇总结果及历史快照
- Gateway 扩展 `/wealth/**` 路由，注入 `X-User-Id`/`X-Family-Id`

### 1.2 范围

| 类别         | 内容                                                                                                    |
|--------------|---------------------------------------------------------------------------------------------------------|
| **前端**     | 账户管理页（`/wealth/accounts`）、资产趋势页（`/wealth/history`）；Dashboard 展示实时个人/家庭资产卡片；wealthStore 新增；API 封装 |
| **后端**     | 新建 `house-hold-wealth` 微服务；账户 CRUD；资产汇总接口；快照生成与历史查询；Gateway 新增路由         |
| **数据库**   | `account` 表、`wealth_snapshot` 表；`account` 按 `user_id` 预留分表扩展                                  |
| **缓存**     | 账户列表（L1+L2）、个人/家庭汇总（L1+L2）、历史快照列表（L2）；缓存失效与防击穿/穿透/雪崩策略         |

### 1.3 不纳入本迭代

- 账户变更流水（记账/收支明细）
- 预算管理
- 报表导出
- 用户分表实现（预留字段，不做分库）
- 定时任务批量快照（本迭代由写操作触发实时快照）

### 2.2 账户管理页（`/wealth/accounts`）

**页面结构**
```
[当前总资产: ¥ xxx] [刷新]
[新增账户 按钮]

账户列表（卡片 or 表格）：
  - 账户名称  类型标签  余额  货币  操作(编辑/删除)

[新增/编辑账户 弹框/展开表单]
  账户名称*
  账户类型*（下拉：储蓄卡/信用卡/股票/基金/支付宝/微信/其他）
  余额*（数值，信用卡填当前账单金额）
  货币（默认 CNY）
```

**交互行为**
- 进入页时调 `GET /wealth/accounts` 拉取列表；同时展示底部汇总 `GET /wealth/summary/user`。
- 新增/编辑表单提交后刷新列表与汇总。
- 删除需二次确认。
- 信用卡余额以红色（负债）展示，其余以蓝色（资产）展示。
- 汇总规则：总资产 = Σ 非信用卡余额 − Σ 信用卡余额。

**字段校验**
- 账户名称：必填，1-64 字符
- 账户类型：必填
- 余额：必填，数值，≥ 0（信用卡账单同）
- 货币：选填，默认 CNY

### 3.3 数据库设计

#### 3.3.1 account 表

```sql
CREATE TABLE IF NOT EXISTS account (
  id          BIGINT PRIMARY KEY          COMMENT '账户 ID（应用生成）',
  user_id     BIGINT NOT NULL             COMMENT '所属用户 ID',
  family_id   BIGINT DEFAULT NULL         COMMENT '所属家庭 ID（冗余，便于家庭维度查询）',
  account_name VARCHAR(64) NOT NULL       COMMENT '账户名称',
  account_type VARCHAR(32) NOT NULL       COMMENT '类型: SAVING/CREDIT_CARD/STOCK/FUND/ALIPAY/WECHAT/OTHER',
  balance      BIGINT NOT NULL DEFAULT 0  COMMENT '余额（单位：分，避免浮点精度问题）',
  currency     VARCHAR(8) NOT NULL DEFAULT 'CNY'  COMMENT '货币',
  created_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at   DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  KEY idx_user_id   (user_id),
  KEY idx_family_id (family_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='个人账户表';
```

**设计说明**
- `balance` 存为整数（分），避免浮点精度问题，前端展示时 ÷100。
- 信用卡账单也用此字段，汇总时按 `account_type = 'CREDIT_CARD'` 区分处理（作为负债相减）。
- `family_id` 冗余存储，避免家庭汇总时跨服务查询用户。

#### 3.3.2 wealth_snapshot 表

```sql
CREATE TABLE IF NOT EXISTS wealth_snapshot (
  id            BIGINT PRIMARY KEY         COMMENT '快照 ID',
  owner_type    VARCHAR(16) NOT NULL       COMMENT 'USER 或 FAMILY',
  owner_id      BIGINT NOT NULL            COMMENT '用户 ID 或家庭 ID',
  total_assets      BIGINT NOT NULL DEFAULT 0  COMMENT '总资产（分）',
  total_liabilities BIGINT NOT NULL DEFAULT 0  COMMENT '总负债（信用卡账单，分）',
  net_worth         BIGINT NOT NULL DEFAULT 0  COMMENT '净资产 = total_assets - total_liabilities（分）',
  snapshot_date DATE NOT NULL              COMMENT '快照日期（按天聚合）',
  created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY uk_owner_date (owner_type, owner_id, snapshot_date),
  KEY idx_owner (owner_type, owner_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='财富快照表';
```

**设计说明**
- `UNIQUE KEY (owner_type, owner_id, snapshot_date)` 保证同一天同一主体最多一条，`INSERT ... ON DUPLICATE KEY UPDATE` 实现幂等更新。
- 按天聚合而非每次操作都插新行，趋势图数据量可控。

### 3.7 核心业务逻辑

#### AccountService

```
createAccount(userId, familyId, req):
  1. 生成 ID（AtomicLong / 雪花算法占位）
  2. 保存 account
  3. 清除账户列表缓存（用户维度）
  4. 触发快照生成（用户 + 家庭）
  5. 返回 AccountResponse

updateAccount(userId, accountId, req):
  1. 查询 account，校验 account.userId == userId（否则 403）
  2. 更新字段并 save
  3. 清除账户列表缓存
  4. 触发快照生成
  5. 返回 AccountResponse

deleteAccount(userId, accountId):
  1. 查询并校验所有权
  2. 删除 account
  3. 清除账户列表缓存
  4. 触发快照生成
  5. 204

getAccounts(userId):
  1. 读缓存（AccountCacheService）→ 若命中则返回
  2. 从 DB 查 findByUserId，写缓存，返回
```

#### SnapshotService — 快照触发逻辑

```
triggerSnapshot(userId, familyId):
  用户快照：
    accounts = accountRepo.findByUserId(userId)
    totalAssets      = Σ balance（非信用卡）
    totalLiabilities = Σ balance（信用卡）
    netWorth         = totalAssets - totalLiabilities
    upsertSnapshot(USER, userId, today)

  家庭快照（若 familyId != null）：
    allAccounts = accountRepo.findByFamilyId(familyId)
    // 同上汇总逻辑，面向家庭全部成员
    upsertSnapshot(FAMILY, familyId, today)

  清除汇总缓存（用户 + 家庭）
```

#### SummaryService — 实时汇总（不依赖快照）

```
getUserSummary(userId):
  1. 读 SummaryCacheService
  2. 未命中：从 DB 实时计算账户列表汇总，组装 WealthSummaryResponse
  3. 写缓存（TTL 60s，短 TTL 保证实时性）
  4. 返回

getFamilySummary(familyId):
  类似，按家庭维度汇总
```

## 四、缓存策略

### 4.1 缓存对象与 Key

| 缓存对象         | Key 格式                              | L1 TTL | L2 TTL         | 失效时机                     |
|------------------|---------------------------------------|--------|----------------|------------------------------|
| 账户列表（用户） | `household:wealth:accounts:user:{uid}`| 60s    | 300s + rand(60)| 新增/更新/删除账户            |
| 账户列表（家庭） | `household:wealth:accounts:family:{fid}` | 60s | 300s + rand(60)| 任意成员账户变更              |
| 个人汇总         | `household:wealth:summary:user:{uid}` | 30s    | 60s + rand(30) | 账户变更触发（短 TTL 容忍略滞后）|
| 家庭汇总         | `household:wealth:summary:family:{fid}`| 30s   | 60s + rand(30) | 任意成员账户变更              |

历史快照：直接查 DB（快照数据不频繁变动，不做 L1，Redis TTL 600s+rand(120)）。

### 4.2 防击穿、穿透、雪崩

与迭代二策略一致，wealth-service 复用相同模式：

- **防击穿**：`ReentrantLock` 单飞（单节点），后续可换 Redisson 分布式锁。
- **防穿透**：不存在数据写入 `__NULL__` 占位，TTL 120s。
- **防雪崩**：Redis TTL = baseTTL + `ThreadLocalRandom.current().nextInt(maxRand)`。

### 4.3 汇总缓存一致性

账户变更 → 删除汇总缓存（而非更新），下次请求时重新计算写入。采用**失效模式（Cache-Aside / Delete on Write）**，保证最终一致，容忍秒级延迟。
